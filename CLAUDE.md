# CLAUDE.md

减少常见 LLM 编码错误的行为准则。可根据需要与项目特定说明合并使用。

**权衡：** 以下准则偏向谨慎而非速度。对于琐碎任务，请自行判断。

## 0. 沟通要求

a. 解释默认中文，代码、技术术语、学术引用使用精准的英文。
b. 直接给出代码结果和答案，仅在必要处加注释。
c. 省略道歉、"我明白"等语气词。
d. 精准优于友好。

## 1. 先思考，再编码

**不要假设。不要隐藏困惑。暴露权衡。**

实现之前：
- 明确陈述你的假设。如果不确定，请提问。
- 如果存在多种解读，请一并提出——不要默默选择。
- 如果存在更简单的方法，请说出来。必要时提出反对。
- 如果有不清楚的地方，停下来。指出困惑之处。提问。

## 2. 简洁优先

**用最少的代码解决问题。不做推测性编码。**

- 不实现未被要求的功能。
- 不为一次性代码创建抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能出现的场景做错误处理。
- 如果你写了 200 行而它本可以只用 50 行，请重写。

问问自己："资深工程师会觉得这过于复杂吗？"如果是，请简化。

## 3. 手术式修改

**只动你必须动的。只清理你自己的遗留问题。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的东西。
- 匹配现有风格，即使你会有不同的做法。
- 如果发现无关的死代码，提出来——但不要删除它。

当你的修改产生遗留时：
- 删除因你的修改而不再使用的导入/变量/函数。
- 除非被要求，否则不要删除预先存在的死代码。

检验标准：每一行改动都应能直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环验证。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后使测试通过"
- "修复 bug" → "编写可复现 bug 的测试，然后使测试通过"
- "重构 X" → "确保重构前后测试均通过"

对于多步骤任务，简要陈述计划：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

强有力的成功标准让你能够独立迭代。薄弱的标准（"把它搞定"）则需要不断澄清。

---

**这些准则生效的标志是：** diff 中不必要的修改更少，因过度复杂化导致的重写更少，澄清性问题出现在实现之前而非错误之后。

## 5. 作用域约束

**本项目所需的一切（含 python 虚拟环境）均在本项目目录内。严禁对本项目目录外的任何文件进行增、删、改操作。**

## 6. 提交规范

提交信息格式：`<TYPE>(<子项目>): <描述>`

**TYPE**
- `FEAT` — 新功能
- `FIX` — 修复 bug
- `REFACTOR` — 重构
- `DOCS` — 文档变更
- `PERF` — 性能优化
- `CHORE` — 杂项维护

**子项目** 取根目录下子文件夹名，如 `get_buildings`、`get_city_poly`、`get_skyline_area` 等。

示例：
```
FEAT(get_buildings): add multi-threaded footprint extraction
FIX(get_city_list): correct WGS84 coordinate offset
REFACTOR(skyline_para_cal): simplify sampling logic
```

不涉及具体子项目时省略括号：
```
CHORE: update .gitignore
DOCS: add API reference
```

## 7. 虚拟环境优先

**运行任何代码一律优先使用本项目下的虚拟环境，而非系统默认环境。**

- 执行 Python 代码前，检查本项目目录下是否存在 `.venv`、`venv` 或其他形式的虚拟环境，若有则激活后再运行。
- 若本项目尚无虚拟环境，不得使用系统全局 Python 环境执行代码，应先在项目内创建虚拟环境。
- 使用 `pip install` 安装依赖时，确保已激活项目虚拟环境，不得安装到系统全局 site-packages。
